## 背景・狙い

環境課との相談で出た「カメラで読み取って**辞書に無かったとき**の対応」への回答。

- **ユーザー価値**: 「見つからない」で終わらせず、「報告する」ボタンで担当者に伝わり、「すみません、すぐ対応します」と返せる安心感を出す。
- **行政アピール価値（本命）**: 辞書に無い品目＝**飯田市公式データ（さんあ〜る／カレンダー）の穴**そのもの。報告が貯まると「市民が調べたのに公式に無かった品目リスト」になり、市へ還元できる。**「補助金でなくリリースしたい／鍵は公式データの利用許諾」という 02 の軸に対し、"データ整備に貢献するツール" という立て付けの交換材料**になる。

## 設計判断

- **自動送信はしない＝ボタン方式**。本アプリ最大の売りが「プライバシー最優先（自動収集ゼロ）」で、プライバシーポリシーにも「操作履歴を収集しない」と明記しているため。ユーザーが押したときだけ、押した品目名のテキストだけを送る。
- **送る情報はテキストのみ**: AI 判定の品目名・地区名・任意コメント・発生元（camera/search）。**画像・位置情報・元デバイス ID は送らない**（`X-Device-Id` はレート制限用ハッシュのみ）。
- **対象は「AI が品目名を返したが辞書に無い」ケース（`unknown_name`）**。「AI が品目名すら出せない」ケース（`not_identifiable`）は公式データの穴ではなく AI 精度の問題なので v1 では対象外。
- 転送先は Worker の `REPORT_WEBHOOK_URL`（Discord 互換 Webhook）。未設定でも受理する（転送しないだけ）→ 運用者のスマホへ即通知でき、デモで「ボタン→通知が届く」を見せられる。

## 関連メモ

- フィードバック機能（B/C）は「環境課の反応を見て実装予定」（CLAUDE.md）だったが、デモで見せられる形が交渉を後押しすると判断し先行実装。
- 自動ログ（全件の受動収集）にすればデータ網羅性は上がるが、プライバシー約束（操作履歴を収集しない）と矛盾するため採用しない。網羅性が本当に必要になった段階で、ポリシー改定とセットで再検討する。
